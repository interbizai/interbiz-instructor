## 2. 코드 측 동시성 안전성 감사

### 🔴 발견된 위험 4가지

#### R1. Race Condition — 평가 저장 동시 시도

**상황**: 강사 A 와 관리자 B 가 동시에 같은 강사의 평가 저장
**현재 코드**: `last write wins` (마지막 저장이 덮어씀)
**위험**: 데이터 손실 가능
**해결**: `updated_at` 컬럼 + Optimistic Concurrency Control

#### R2. 동시 사진 업로드 시 Storage upsert 충돌

**상황**: 50명이 동시에 본인 사진 업로드
**현재 코드**: `storage.upload(path, ..., {upsert:true})` → race 시 일부 실패 가능
**위험**: 1~2명 사진 등록 실패 (silent retry 없음)
**해결**: 재시도 로직 (지수 백오프 3회)

#### R3. 인증 만료 동시 폭주

**상황**: 50명이 동시에 24h 토큰 만료 시점에 접속
**현재 코드**: 각자 401 받고 동시에 재로그인 시도 → /api/auth/login 동시 호출 50회
**위험**: bcrypt.compare 50회 동시 → CPU 한계
**해결**: 클라이언트 측 토큰 만료 임박 시 사전 갱신 (refresh token)

#### R4. 캐시 무효화 폭주

**상황**: 한 명이 평가 등록 → `bumpDataVersion()` → 다른 사용자 캐시 무효화 동기 못 함
**현재 코드**: 클라이언트별 캐시 독립
**위험**: 일부 사용자 화면이 5분간 stale 데이터 표시
**해결**: Supabase Realtime 구독 (다른 사용자 변경 즉시 인지)

### 🟠 발견된 개선점 6가지

#### I1. 가상화 미적용 — 관리자 강사 테이블

50명+ 데이터 → 모든 행 한 번에 DOM. 스크롤 끊김. 메모리 부담.

#### I2. 이벤트 핸들러 인라인 — 메모리 부담

각 행마다 `onclick="..."` 문자열 → 50명 × 5개 버튼 = 250개 함수.

#### I3. 무거운 계산 메모이제이션 없음

`getFiltered()` 가 페이지 진입·필터 변경 때마다 50명 재계산.

#### I4. 백업·복구 자동화 없음

DB 사고 시 복구 가이드 없음.

#### I5. 사용자 측 에러 모니터링 없음

강사가 에러 만나도 우리는 모름 (사용자가 알려야).

#### I6. 영상 분석 동시 폭주 제한 없음

50명이 동시 "AI 분석 시작" 클릭 시 Vertex AI 한도 초과.
