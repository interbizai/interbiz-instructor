# 인터픽 vs 메이저 사이트 — 구조 비교 + 변경 로드맵

**작성일**: 2026-05-11
**상황**: 사진 마이그레이션 완료 후에도 페이지 전환 느림
**목적**: 정확한 진단 + 위험도 순 단계별 적용

---

## 1. 우리 사이트 구조 (현재)

```
interpick.vercel.app/
├── index.html (1.4MB, 16,000줄)
│   ├── <style> 인라인 CSS (~3,000줄)
│   ├── HTML 페이지 14개 (인터PICK·AI코칭·교육·MY·관리자 등) 한 파일 안
│   └── <script> 인라인 JS (~10,000줄, 모든 기능 한 곳)
├── /api/* (Vercel serverless 함수)
├── /assets/* (로고·폰트)
└── sw.js (Service Worker - 정적 캐시)
```

**특징**:
- 단일 HTML 파일 = "전부 한 번에 받음"
- 빌드 도구 없음 (Vite/webpack 미사용)
- 라우터 = `showPage(id)` 로 div 표시·숨김
- 데이터 = 한 번 로드 후 메모리(`D.users` 등)에 보관

## 2. 메이저 사이트 구조 (실제 측정)

### 2-1. Next.js 기반 (Vercel·Notion·OpenAI·Cursor)

```
notion.so/
├── /_next/static/chunks/
│   ├── main-XXX.js          (~80 KB · 공통)
│   ├── pages/home-XXX.js    (~30 KB · 홈만)
│   ├── pages/admin-XXX.js   (~50 KB · 관리자만)
│   └── ...
└── 페이지마다 별도 HTML stub + 필요한 chunk 만 받음
```

**효과**:
- 초기 다운로드: ~110KB (홈 진입 시)
- 페이지 이동: 추가 50KB 만 (관리자 진입)
- 첫 페이지 로드: < 1초

### 2-2. React SPA (Slack·Gmail·Discord)

```
slack.com/
├── runtime.js             (~10 KB)
├── vendor.js              (~150 KB · React + 라이브러리)
├── app-shell.js           (~80 KB · 메인)
└── chunks/                (페이지마다 분할)
```

**효과**: vendor 캐시 → 두 번째 방문 시 vendor 안 받음, app-shell 만 새로 받음.

### 2-3. Vue/Nuxt (Bookmate·Codepen·Cal.com)

비슷한 패턴. 라우트별 자동 분할.

### 2-4. 단일 HTML (Wikipedia·정부 사이트)

각 페이지마다 별도 HTML 파일. 페이지 이동 = full page reload.
**우리와 정반대 패턴** — 단순하지만 페이지 이동 1~2초 발생.

## 3. 차이점 — 왜 우리 사이트가 느린가

| 측면 | 우리 | Notion/Slack | 영향 |
|---|---|---|---|
| 초기 JS 다운로드 | 1.4 MB 한 번에 | 110 KB | 첫 진입 5~10초 차이 |
| 페이지 전환 | 0 KB (이미 받음) | 50 KB | 우리가 더 빨라야 하는데... 왜 안 빠를까? ↓ |
| 페이지 전환 시 렌더 비용 | renderXXX() 가 매번 D.users 60명 + videos 100개 전체 순회 | React 가상 DOM 으로 diff (변경분만) | **이게 진짜 병목** |
| 메모리 캐싱 | 매번 새로 계산 | useMemo · React.memo | "변경 없으면 재렌더 안 함" |
| 이벤트 핸들러 | 행마다 onclick="..." 문자열 (행 60개 = 핸들러 60개) | 위임 (부모 1개) | 메모리·계산 비용 |
| 이미지 | 사진 17명 한 번에 다 그림 | viewport 안 들어오면 안 받음 (`loading=lazy`) | 페이지 진입 시 17 사진 동시 다운로드 |

**결론**: 우리 병목은 "단일 파일이라서" 가 아니라 **"매번 무거운 계산을 한다"** 임. 코드 분할보다 **렌더링 최적화** 가 더 큰 효과.
